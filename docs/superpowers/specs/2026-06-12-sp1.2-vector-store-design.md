# SP1.2 ‚Äî VectorStore Abstraction + LanceDB Backend

**Date:** 2026-06-12 ¬∑ **Status:** approved (autonomous mandate)
**Motivation:** The north-star architecture puts vectors in LanceDB (`chunk_vectors`, `entity_vectors` keyed by Postgres ids) while Postgres stays system-of-record. Today embeddings live in pgvector columns (`chunks.embedding`, `entities.embedding`, 768-dim, HNSW). SP1.2 introduces the seam: a `VectorStore` interface with two backends, switched by env flag, default unchanged.

## Decisions (locked)

1. **Interface + two backends** ‚Äî `VectorStore` ABC; `PgVectorStore` (SQL over the existing columns ‚Äî today's behavior) and `LanceDBStore` (embedded LanceDB tables). Rejected: hard cutover to LanceDB (breaks the running stack, loses pgvector HNSW fallback); dual-write (two sources of truth).
2. **Flag** ‚Äî `VECTOR_BACKEND=pgvector|lancedb`, default `pgvector` (additive convention: default = today's behavior, byte-for-byte). `LANCEDB_URI` default `./data/lancedb`.
3. **All embedding reads AND writes route through the store.** No call site touches `embedding` columns directly anymore. In `lancedb` mode the PG columns simply stay NULL; bookkeeping columns (`embedding_model`) stay in PG either way.
4. **Hit-then-hydrate** ‚Äî `search_*` returns `(id, distance)` pairs; callers hydrate rows from Postgres by id. Uniform across backends ‚Üí parity by construction. (pgvector pays one extra `WHERE id IN` query; negligible at our QPS.)
5. **Raw-vector getter** ‚Äî `get_entity_vectors(ids)` serves the Python-cosine consumers (EntityResolutionService scoring, LinkingService hybrid score), so resolution works in both modes.
6. **Migration tool, both directions** ‚Äî `scripts/migrate_vectors.py --to lancedb|pgvector` copies whichever side has data; idempotent, batched.

## Interface

```python
class VectorHit(NamedTuple):
    id: int          # chunks.id / entities.id are int PKs
    distance: float  # cosine distance, lower = closer

class VectorStore(ABC):
    backend: str
    async def upsert_chunks(self, items: Sequence[tuple[id, source_id, list[float]]]) -> None: ...
    async def upsert_entities(self, items: Sequence[tuple[id, list[float]]]) -> None: ...
    async def search_chunks(self, vec, *, limit: int, source_id=None) -> list[VectorHit]: ...
    async def search_entities(self, vec, *, limit: int) -> list[VectorHit]: ...
    async def get_entity_vectors(self, ids) -> dict[id, list[float]]: ...
    async def delete_chunks_for_source(self, source_id) -> None: ...
    async def delete_entities(self, ids) -> None: ...
```

Factory `get_vector_store(settings) -> VectorStore` in the same module; LanceDB connection cached per URI. Services accept `vector_store=None ‚Üí get_vector_store(settings)` so existing construction sites keep working.

## Backend mechanics

**PgVectorStore** ‚Äî `upsert_*` = `UPDATE ... SET embedding = :v` (exactly today's writes); `search_*` = `SELECT id, embedding <=> CAST(:v AS vector) AS d ... WHERE embedding IS NOT NULL [AND source_id=:s] ORDER BY d LIMIT :k` (today's queries minus the joins, which move to hydration); deletes = no-op (row lifecycle owns the column).

**LanceDBStore** ‚Äî `lancedb.connect_async(uri)`; tables `chunk_vectors(id: str, source_id: str, vector: FixedSizeList[float32, dim])`, `entity_vectors(id: str, vector)`; dim from `settings.embedding_dimensions`. Upsert = `merge_insert("id")`; search = `vector_search(vec).distance_type("cosine").limit(k)` with `source_id` prefilter; deletes = `tbl.delete("...")`. Lazy table creation. Exact API verified against the INSTALLED lancedb version.

## Error handling

- LanceDB URI unreachable/corrupt ‚Üí store raises at first use; ingest step fails visibly (same contract as PG outage). No silent fallback across backends (would split data).
- Dimension mismatch guarded by existing `_validate_embedding_dimensions` upstream; LanceDB schema enforces fixed-size list as second line.
- Concurrency note: LanceDB local-FS commits use optimistic concurrency; our write paths are worker-only (single process). Documented: `lancedb` mode assumes one writer (worker); backend reads are safe.

## Boundaries

- pgvector columns + HNSW indexes **stay in the schema** (no drop migration in SP1.2; column removal happens only at a future full-cutover SP, per north-star "schema deltas").
- Compose stays `pgvector` default ‚Äî no infra change shipped.
- No re-embedding: migration copies vectors as-is.
